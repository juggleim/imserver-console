## Context

推送设置页目前用 `settings` 元数据描述 9 个渠道，但每个渠道只绑定一个 `ref` 表单。页面切换标签时调用单记录查询接口，保存时 Android 厂商写入 `androidpushconfs`，FCM 复用该表，iOS 写入 `ioscertificates`。DAO 的查询方法同样只返回一条记录，现有 upsert 依赖数据库唯一键，因此同一应用、同一渠道的新包名可能覆盖旧包名。

本变更横跨 Vue 页面、文件上传、Go handler/model/DAO 和数据库索引。既有配置必须在升级后自然显示为一张卡片；密钥和证书密码不能因列表化展示而泄露，也不能在仅修改其他字段时被掩码占位值覆盖。

## Goals / Non-Goals

**Goals:**

- 让每个现有推送渠道按包名维护任意多组配置，并以卡片方式新增和编辑。
- 复用一套渠道元数据驱动卡片字段、弹窗字段、校验和请求转换，避免为每个厂商复制交互逻辑。
- 为普通 JSON 配置和 iOS/FCM 文件型配置提供一致的列表身份与编辑语义。
- 保留既有数据，明确重复包名、包名修改、敏感字段和文件保留规则。

**Non-Goals:**

- 不新增或删除推送厂商，也不改变各厂商凭证字段含义。
- 不提供配置删除、启停、排序、批量导入或跨渠道复制能力。
- 不改变消息发送时按包名选择推送配置的业务逻辑。
- 不重做整个应用管理页面或音视频配置页。

## Decisions

### 1. 以 `(app_key, push_channel, package)` 作为配置业务身份

Android 厂商和 FCM 在 `androidpushconfs` 中以应用、规范化渠道名和包名唯一；iOS 在 `ioscertificates` 中以应用和包名唯一。列表按稳定顺序返回，至少以包名升序保证刷新后卡片顺序一致。

选择该方案是因为运行时已有按包名查询的意图，且独立行能继续承载 FCM/iOS 二进制证书。备选方案是在单行 `push_conf` 中保存数组，但这会放大单次更新、妨碍按包名检索并使证书存储模型分裂，因此不采用。

### 2. 增加显式列表查询，保存接口支持原始包名

为 Android/FCM 和 iOS 提供按应用及渠道返回数组的列表能力，页面不再依赖旧的单记录查询。现有保存/上传请求增加可选 `original_package`：新增时为空；编辑时携带打开弹窗时的包名。服务端在事务内检查目标包名是否冲突，再更新精确的原配置；包名未变化时执行普通更新。

保留旧的单记录读取入口和既有记录解析，避免其他调用方立即失效。备选方案是直接把当前 `/get` 响应从对象改为数组，但这属于不必要的破坏性变更，因此不采用。

### 3. 渠道元数据同时驱动列表、弹窗和安全展示

前端把当前每渠道单一 `state` 改为 `items` 列表，并在渠道字段元数据中补充 `required`、`secret`、`cardVisible`、文件类型和选项信息。新增和编辑使用独立的弹窗草稿，打开时深拷贝，不直接修改卡片数据；保存成功后重新获取当前标签列表，以服务端结果作为唯一事实源。

每个标签始终提供新增卡片：空列表时页面只有新增卡片；有配置时配置卡片与新增卡片并列。配置卡片标题为包名，正文按该渠道实际字段展示，底部只有“设置”操作。

备选方案是在卡片内原地编辑，但文件输入、取消恢复和跨渠道字段差异会让状态管理复杂，也不符合需求指定的弹窗交互，因此不采用。

### 4. 密钥类字段仅掩码展示，空值表示编辑时保留

App Secret、Master Secret、证书密码等字段在列表响应和卡片中只返回/展示统一掩码，不返回原始值。编辑弹窗中的密钥输入保持为空并提示“留空则不修改”；服务端在编辑请求对应字段为空时保留原值，在新增请求中仍执行必填校验。前端永远不把掩码字符串提交为真实值。

此方案牺牲了查看原密钥的便利性，但降低了管理控制台泄露凭证和误覆盖密钥的风险。

### 5. 文件型渠道沿用专用上传接口并支持保留旧文件

iOS 卡片展示证书文件名、VoIP 证书文件名和环境，FCM 卡片展示配置文件名。编辑时未选择新文件表示保留数据库中的原文件和文件名；选择新文件才替换对应内容。新增配置则要求满足该渠道现有文件必填规则。普通文本保存与 multipart 上传均使用相同的 `original_package` 语义。

### 6. 兼容迁移先改唯一键，再发布应用代码

数据库迁移将 Android/FCM 表的唯一约束调整为应用、渠道、包名组合，将 iOS 表的唯一约束调整为应用、包名组合，并保留已有行。代码发布后，旧行会由列表接口直接返回，无需数据内容转换。

若不同环境的实际索引名或现有约束与预期不一致，迁移脚本必须先探测 schema 并在测试环境验证；不能只依赖应用层重复检查来保证唯一性。

## Risks / Trade-offs

- [数据库索引定义不在当前仓库中，环境之间可能不一致] → 实现前确认实际 DDL，把可重复执行的索引迁移随发布包交付，并在预发检查组合唯一键。
- [编辑包名时发生并发冲突或只更新了部分字段] → 使用事务、数据库组合唯一键和冲突错误映射，更新条件同时包含应用、渠道和原始包名。
- [列表返回大量二进制证书或原始密钥导致性能与安全问题] → 列表 DTO 排除证书字节并掩码敏感字段，只返回文件名和展示所需元数据。
- [用户把空密钥误解为清空密钥] → 弹窗明确提示“留空则不修改”；本变更不提供清空凭证语义。
- [只刷新前端本地数组会与服务端规范化渠道名或文件名不一致] → 每次保存成功后重新请求当前标签列表。
- [新增卡片与配置卡片在窄屏下拥挤] → 复用音视频配置的响应式卡片网格并为弹窗字段提供单列布局。

## Migration Plan

1. 核对目标环境中 `androidpushconfs` 和 `ioscertificates` 的现有主键/唯一索引及重复数据。
2. 备份相关表，处理会阻止组合唯一键创建的重复行，并执行可回滚的索引迁移。
3. 发布包含 DAO、列表接口、兼容保存语义和前端卡片页面的应用版本。
4. 在预发验证既有单配置显示、多包新增、包名修改、密钥保留、iOS/FCM 文件保留和推送运行时按包名读取。
5. 若需回滚应用代码，先停止新增多包数据；旧版本只会读取其中一条，因此数据库索引回滚必须在确认并归并多余记录后单独执行，不能直接删除用户配置。

## Open Questions

- 实现阶段需从部署环境或上游数据库仓库确认两个表的真实 DDL、索引名称及数据库版本，以确定最终迁移脚本语法。
- 如果产品后续要求删除配置或主动清空密钥，需要另行定义确认交互、审计与服务端语义；本次不包含。
